## Scenario

**Brightline Logistics** *(fictional company)* gets the same handful of policy questions every week — parental leave, remote-work eligibility, benefits enrolment windows, expense rules. HR answers them by hand, answers drift depending on who replies, and employees wait. Leadership has asked you to design (on paper) an **HR Policy Agent** before any tool is configured. There is no single correct design — you must make defensible choices and justify them.

## Exercise 1 — Specify a safe HR Policy Agent

**⏱️ Estimated time: 20 minutes**

### Context

A good agent spec is mostly about restraint: what it will NOT do is as important as what it will. You are writing the document an IT or compliance partner could review before anything is built.

### What You Will Practice
- Scoping an agent narrowly (purpose)
- Writing explicit instructions, grounding, and boundaries
- Designing escalation and post-launch monitoring (independent research encouraged)

### Instructions

1. Write a **one-sentence purpose** that keeps the agent's role narrow, and explain in one line why a narrow scope is safer here.
2. Define the four building blocks for this agent: **purpose, instructions, grounding source(s), and boundaries.** Be specific — name the document(s) it should be grounded in.
3. List **at least three** questions the agent should answer and **at least three** it must refuse and escalate. Write the exact **escalation message** employees would see when it hits a boundary.
4. Write a short **operating plan**: the five monthly-review checks you would run, the four signals you would monitor, and what you would do the day a policy changes.

### Research Pointers

You may want to look at how Microsoft frames the "from prompts to agents" pattern, and how a 30-minute monthly review and a four-week rollout (test 20 questions → pilot with 10–15 people → update → set review reminders) are used to catch drift before it spreads.

### Deliverable Format

A one-page agent specification, structured as:

| Section | Your specification |
|---|---|
| Purpose (1 sentence) + why narrow | |
| Instructions (what it can / cannot do) | |
| Grounding source(s) | |
| Boundaries (when to stop) | |
| 3+ questions it answers | |
| 3+ questions it refuses + escalation message | |
| Operating & monitoring plan | |
